## 2026-07-07 ‚Äî Spec v1 revisions

  `@id`, `_ <-` for discard, explicit `return { ... }` blocks. Multi-line

  so v1.1 dynamic-workflow evaluation can reuse it at runtime.

## 2026-07-07 ‚Äî Revised: skip `LLM.Load`, own key store

Reconsidered the `llm-simple` `Load` decision after user push-back.
`LLM.Load.loadGateways` combines three concerns we want separated:
`.env` discovery via cwd, key lookup from hardcoded env-var names, and
gateway construction. Using it forced us into:

- Chdir at startup (surprising, affects unrelated CLI path resolution).
- `.env` pinned to project dir (couples secrets to source tree).
- Partial `error` + silent `catch SomeException` in the wrappers.
- Inconsistency with our own `Secret<T>` discipline (keys enter as
  opaque closures, never as typed secrets).

**New design (spec ¬ß7.2, ¬ß7.3 rewritten):**

- `hwfi` uses `LLM.Providers.*` gateway constructors and
  `LLM.Load.ModelCatalog.loadModelCatalog` directly. Skips
  `LLM.Load.loadGateways` and the `*OrThrow` model loaders.
- Own `KeyStore :: Map ProviderName (Secret Text)`, populated by
  merging (highest first) `--env-file`, `<project>/.env`,
  process environment. Uses `Configuration.Dotenv.parseFile` (no
  process-env injection). No chdir.
- Every project **must** ship its own `model-catalog.json`. No engine
  default. Simpler, more explicit, avoids "our default changed"
  regressions. Dropped backlog task 6.8 (catalog merging); replaced
  with 6.8 for a user-level `$XDG_CONFIG_HOME/hwfi/.env` fallback.
- Provider‚Äìkey linkage validated at startup against catalog contents.
  Missing key fails immediately with a message naming the offending
  model and provider (new A12).
- Added `--env-file` CLI flag (¬ß9). Removed the cwd/chdir language and
  the "engine default catalog" language from ¬ß7.
- TASKS 1.6 rewritten as `Hwfi.Runtime.KeyStore`; 1.7 rewritten to
  require project-local catalog; 4.5 rewritten as
  `Hwfi.Runtime.Gateways` on top of `LLM.Providers.*`.

## 2026-07-07 ‚Äî M3 type checker complete

- Milestone: `hwfi check` type-checks a whole project into a
  `TypedProject` (resolved signatures, per-step cacheability, Merkle
  fingerprints). 71 tests green across unit specs and expected-error
  fixtures (`test/fixtures/check/`).
- Decision: split the type world in two ‚Äî the surface `TypeExpr` (AST,
  unresolved, alias refs allowed) vs `Hwfi.Type.Type` (resolved, used by
  the checker). Aliases are expanded during resolution; cycles are an
  error (A10).
- Decision: assignability is `structEq` plus one subtyping rule ‚Äî a
  `String` is accepted where a `FileRef` is expected ‚Äî so literal path
  strings can feed file-taking parameters without a cast. Records compare
  structurally (field order irrelevant); lists/secrets are congruent.
- Decision: declaration `Fingerprint` is a Merkle hash (crypton SHA256)
  over the canonical AST encoding plus the sorted callee fingerprints,
  taken over the acyclic direct call graph. Builtins are leaves with a
  fixed identity; bare first-class refs are not static callees.
- Blocker resolved (root cause): `computeFingerprints` hung on any
  multi-decl project. Cause: `Fingerprint` is a `newtype`, so forcing a
  value to WHNF forces its hash and therefore its callee lookups into the
  self-referential `fps` map; `Data.Map.Strict.mapWithKey` forces every
  value during construction, demanding a callee's fingerprint before
  `fps` exists ‚Üí loop. Fix: build `fps` with `Data.Map.Lazy.mapWithKey`
  (tie-the-knot), leaving lookups on the shared map type unchanged.
- Decision: checker is pure (`Project -> Either [TypeError] TypedProject`),
  keeping IO in the CLI only ‚Äî required for the deferred v1.1 dynamic
  workflow evaluation.

## 2026-07-07 ‚Äî M5 persistence, tracing, resume complete

- Milestone: `hwfi run` persists a run under `<workspace>/.hwfi/runs/<id>/`
  (`run.json`, `steps/<step-key>.json`, append-only `trace.jsonl`);
  `hwfi resume` re-executes skipping cached cacheable steps; `hwfi show`
  pretty-prints. 128 tests (was 102); A4/A7/A13/A15 covered.
- Decision: the `Tracer` gained an optional file sink ‚Äî `emit` appends the
  event's JSON line and `hFlush`es, so a killed process leaves a durable
  prefix. `eventFromJson` is the exact inverse of `eventToJson`, so resume
  reconstructs `ctx.trace` and continues `seq` from the persisted tail, and
  `hwfi show` decodes the same shape. Same ADT, one seam.
- Decision: step results and root inputs are persisted as **actual**
  (non-redacted) values (cache/`run.json`), because resume must re-evaluate
  with real values; only `trace.jsonl` redacts (¬ß8.3.4, A8). A cached result
  is reconstructed to a typed `RValue` via the step's static result type
  (new `TypedStep.tsResultType`), preserving `FileRef`/`Secret` distinctions
  JSON conflates. The step-key hashes actual (not redacted) secret values so
  distinct secrets get distinct keys without leaking (one-way hash).
- Decision: the cache is consulted **only on resume** but written on every
  attempt. Cacheability is a per-call-site static property: a cacheable step
  that calls a sub-workflow is skipped wholesale on a cache hit even if the
  callee contains non-cacheable steps (its result was persisted). Editing a
  callee changes its Merkle fingerprint ‚Üí the caller's step-key ‚Üí recompute
  (A13); a first-class `Ref` argument contributes its target's fingerprint,
  not its qname (¬ß8.1).
- Decision: `run.json` records `project_dir` so `hwfi resume <ws> <id>` can
  re-parse and re-check the project (needed for A13) without re-supplying the
  path. `project_hash` is now a real content hash over all declaration
  fingerprints (replacing M4's entrypoint-fingerprint stand-in).
- Decision: the workspace lock is an advisory OS `flock` on
  `<ws>/.hwfi/lock`; a second *in-process* open (GHC's single-writer handle
  lock throws `ResourceBusy`) is caught and reported as "locked" rather than
  crashing. The trace reader reads strictly to avoid a lazy read handle
  clashing with the resumed attempt's append handle.

## 2026-07-07 ‚Äî M6 LLM tool-use complete

- Milestone: `builtin/llm-agent` (free-text) and `builtin/llm-agent-object`
  (typed `submit`) drive an agentic tool-use loop. New modules
  `Hwfi.Runtime.{Schema,Agent}`; `Gateways.modelCatalogFingerprint`; agent
  trace events; bespoke agent type-check. 152 tests (was 128); A17‚ÄìA21.
- Decision (scope): implemented a **dedicated agent-loop** reified state
  machine over `RValue` (a round/tool-call machine, modelled on
  `../llm-workflow` but without `unsafeCoerce` since hwfi values are already
  dynamically typed), rather than refactoring the whole executor into a CEK
  machine as `tool-use.md` sketches for M7. Rationale: M6 has one consumer;
  the general control-flow unification is justified only once M7 (`if`/
  `foreach`/`par`) provides the second. Flagged so M7 revisits it.
- Decision (agent eligibility, ¬ß6.1.1/¬ß6.1.5): a declaration may be advertised
  as a tool only if `typeToSchema` succeeds for every input (rejecting
  `Secret<_>`, `ToolRef`/`WorkflowRef`, `Bytes`) and it does not transitively
  reach `builtin/introspect`. Introspect-reachability is a cycle-safe graph
  closure (`Hwfi.Check.reachesIntrospect`) threaded into `CheckCtx`, so the
  check is robust even with import cycles. Agent builtins take a heterogeneous
  `tools` list with no v1 union type, so they bypass the generic `checkArgs`
  path entirely (`checkAgentCall`/`checkToolElem`).
- Decision (caching, ¬ß8.2.1): the agent step is a non-cacheable black box, but
  each **model call** (sub-keyed by agent step-key + round + model-catalog
  fingerprint + advertised-tools fingerprint + canonical messages) and each
  **tool call** (sub-keyed by step-key + round + call index + callee
  fingerprint + canonical args) is content-addressed in `RunStore`. Resume
  consults these sub-caches and replays the model's prior choices and tool
  results without re-paying the LLM or re-running side effects. Verified by an
  end-to-end test that resumes against a gateway/dispatch that fail if touched.
- Decision (tool-result caching): cache the **redacted** JSON of a tool result,
  not the actual `RValue`. Reconstruction shows the model the same redacted
  content it originally saw, and no secret ever reaches the intra-step cache.
  The rare case of an agent tool returning a secret is an accepted limitation.
- Decision (trace ordering): `agent-round-start`/`-round-end` are emitted
  lazily via an `IORef` guard, so a resumed round whose model and tool calls
  are all cache hits emits no new events (¬ß8.3.3.7); the marker is written on
  the first real event of the round. A model tool call is bracketed by a nested
  `step-start`/`step-end` (or `error`) so the callee's events nest under the
  agent step and every start has exactly one terminal.
- Decision (error classes, ¬ß6.1.4): malformed/ill-typed tool arguments, unknown
  tools, callee-surfaced errors, mixed-`submit` rounds, and `submit` schema
  failures are **recoverable** ‚Äî fed back to the model as a tool result so it
  can retry. Only `KInternal` dispatch failures and `max_rounds` exhaustion are
  fatal and abort the step.
- Testing: the agent loop is exercised via a fake `LLMGateway` scripted by
  conversation round (counting `ToolTurn`s), wired into a `ModelConfig` with no
  retries/timeout. `llm-simple` added as a test dependency. Covers free-text
  termination, unknown-tool feedback, `max_rounds` fatality, object/`submit`,
  mixed-submit rejection then retry, schema-invalid submit retry, and
  cached-resume replay (A17‚ÄìA21).

to failures). Specced in spec ¬ß6.2/¬ß6.3/¬ß7.5; design rationale in
tool-use.md ¬ß8; backlog reordered in TASKS.md (M7 now = coding tools,
control flow pushed to M8).

- **Rewrite, don't wrap `llm-simple`'s FS tools.** `llm-simple` has ~15
  `LLM.Tools.*` `TypedTool`s + an executor/loop, but wrapping them was
  rejected: a native `TypedTool` has no Merkle fingerprint (breaks A13
  cache invalidation), runs under `FsConfig`'s weaker canonicalize+re-walk
  sandbox instead of hwfi's lexical `Workspace` guard (A5), and bypasses
  hwfi's `file-io` tracing + secret redaction. Instead, implement mutation
  builtins natively over `Hwfi.Runtime.Workspace`; port only *pure*
  algorithms (glob/regex/binary-detect/find-replace) if worthwhile. One
  sandbox, one trace stream, one fingerprint/cache scheme.
- **New builtins.** Read/nav: `read-file-slice`, `find-files`, `grep`.
  Mutating: `edit-file` (literal find/replace with a required `expect`
  occurrence-count guard, A23), `move-file`, `copy-file`, `remove-file`,
  `make-dir`, `remove-dir`. All cacheable, all through the workspace guard,
  all agent-eligible (they take `FileRef`/`String`).
- **`builtin/exec` posture.** argv (`program` + `args`), **no shell**, so
  no injection surface. Working dir = workspace root; program looked up by
  allowlisted basename on a policy `PATH`. Non-zero exit is a *value*
  (`{ exit_code, stdout, stderr, timed_out }`) so an agent can react, not a
  run abort. Fail-closed: disabled unless `project.json.exec.allow` lists
  the program; empty env except `exec.env`; per-call timeout + output caps.
  New `exec` trace event (¬ß8.3.2). Uses `typed-process`.
- **Resume correctness (the load-bearing argument).** Nothing new: hwfi
  already ships a mutating cacheable builtin (`write-file`), whose A4 rule
  is "a cache hit means the effect is already present in the workspace"
  (durable-workspace invariant, now stated explicitly in spec ¬ß8.2). All
  new mutation builtins inherit it unchanged. `exec` is cached and its
  captured output *replayed* on resume (like a model call, ¬ß8.2.1), so a
  resumed agent loop stays on the same branch and does not re-run a build.
  Step-keys hash args + ctx projection, not live workspace contents ‚Äî the
  same caveat that already governs `read-file`.
- **Not doing OS-level containment in v1.** The allowlist + empty-env is
  the boundary; namespaces/seccomp deferred to M8.2. Documented, not
  silently accepted. Network access via exec'd programs is inherent and
  controlled only by the allowlist (¬ß7.4).
- Added acceptance criteria A22‚ÄìA26; removed "shell/exec tool" from the
  spec ¬ß13 deferred list (replaced with the OS-isolation refinement note).

## 2026-07-07 ‚Äî M7 mutation + exec tools complete

- Milestone: workflows and agents can modify the sandboxed workspace and run
  allowlisted commands. New modules `Hwfi.Runtime.{Glob,Exec}`; navigation +
  mutation ops on `Hwfi.Runtime.Workspace`; `ExecPolicy` in the manifest;
  extended trace `FileOp`/`Exec` events; `execErrors` check pass;

- Decision (grep engine): `regex-tdfa` (pure Haskell, POSIX ERE) rather than a
  PCRE/RE2 C binding, to avoid a new C dependency. Not literal RE2 but adequate
  for the common line-matching patterns `builtin/grep` targets; a malformed
  pattern surfaces as a recoverable `eval` error, not a crash.
- Decision (`edit-file` guard, A23): literal find/replace requires an `expect`
  occurrence count; a mismatch fails the step and leaves the file untouched, so
  an agent cannot silently over-edit. Empty needles are rejected (count is
  undefined). Replaces every occurrence when the count matches.
- Decision (`exec` policy enforcement, A24): a two-layer, fail-closed model.
  `hwfi check` statically rejects a `builtin/exec` whose literal `program` is
  not in `exec.allow`, and rejects `exec` entirely when no policy is configured;
  a *dynamic* program name passes the check and is enforced at runtime by
  `Hwfi.Runtime.Exec` (basename-only, allowlist membership). The child gets only
  the `exec.env`-named variables from hwfi's own environment (never provider API
  keys), a wall-clock timeout, and per-stream output caps.
- Decision (error class of allowlist violations): surfaced as a *sandbox* error.
  Inside the agent loop this is recoverable (the model can retry with an allowed
  program, per ¬ß6.1.4); a direct scripted step aborts. Only internal errors are
  fatal in the loop.
- Decision (durable-workspace resume, A25/¬ß8.2): mutation and `exec` steps are
  cacheable; on resume a completed one is served from cache and its side effect
  is not re-applied (edit) / not re-run (exec, whose captured output is
  replayed). Verified end-to-end in `ExecutorSpec` and per tool call in the
  agent loop (¬ß8.2.1).
- Testing (A26): the end-to-end coding loop runs the real builtins (a genuine

  reads the fed-back `exit_code` from the conversation to decide whether to edit

## 2026-07-08 ‚Äî M8 control flow (`if`/`foreach`/`par`) complete

Milestone: workflow/tool bodies branch and iterate. `Statement` gains
`SIf`/`SLoop`; new parser productions, recursive checker, executor, and trace
events. 207 tests (was 188); new `Hwfi.Runtime.ControlFlowSpec`. Note: spec ¬ß13
listed control flow as *deferred* with no surface-syntax design, so the syntax
and semantics below were designed for this milestone (spec ¬ß13 to be updated).

- **Value-producing constructs, uniform with steps.** All three use the existing
  `binder <- rhs @id` shape. A block's value is its last statement's value (empty
  block ‚Üí empty record). `x <- if ${c} { ‚Ä¶ } else { ‚Ä¶ }` requires `c : Bool`, a
  mandatory `else`, and structurally-equal arm types; `xs <- foreach v in
  ${list} { ‚Ä¶ }` / `par(max = N) ‚Ä¶` bind `List<U>` from a `List<T>` (binding
  `v : T`). Rationale: keep the linear binding model intact rather than
  introducing block-scoped name leakage, and make conditional/loop results
  first-class. Chose this over statement-only (side-effect) control flow because
  the rest of the language is expression/binding-oriented.
- **Flat per-declaration id namespace.** Step binders, loop variables, and
  construct `@id`s must all be unique within a declaration ‚Äî including across
  `if` branches (so sibling arms cannot reuse a binder). No shadowing of outer
  bindings. Simplifies scope + trace correlation and dodges the "which `s`?"
  ambiguity; the cost is uglier names in mirrored branches (accepted).
- **Per-iteration/branch step-key scoping (resume correctness).** The executor
  threads a scope prefix (`check#2/c`, `mode?then/s`) that is folded into the
  step-key hash, so dynamically-distinct occurrences of the same static step get
  distinct cache keys. Decision: thread the scope into sub-workflow calls too
  (call-site-prefixed internal keys), trading some cross-call cache sharing for
  correct per-iteration resume. Empty scope at the top level keeps all existing
  keys/tests stable.
- **`par` semantics.** Bounded concurrency (`pooledForConcurrentlyN`, default 4,
  `par(max = N)` overrides), results ordered by **input index** (not completion),
  aborts on the **lowest-index** failure. Iterations write to distinct cache
  keys/paths (safe concurrently); only the tracer's `emit` critical section is
  serialised via a new `MVar` mutex in `Hwfi.Runtime.Trace`, keeping `seq`
  numbering and on-disk line order consistent under concurrency.
- **Trace.** New `if-branch`/`loop-start`/`loop-iter`/`loop-end` events (JSON
  round-trip + `hwfi show` render). Callee collection, Merkle fingerprint
  encoding, and the `builtin/exec` policy pass all recurse through control-flow
  blocks, so editing a step inside a branch/loop still busts dependent caches and
  a nested un-allowlisted `exec` is still rejected at check time.
- `examples/control-flow`: `par` syntax-check + `foreach` manifest + `if`/`else`
  summary; `hwfi check` clean and ran end-to-end.
