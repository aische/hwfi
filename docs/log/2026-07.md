## 2026-07-07 — Spec v1 pinned

- Turned `idea.md` into `spec.md` with concrete v1 scope.
- Key design decisions locked for v1:
  - Workflow project = directory of `.md` (workflows/tools) + JSON manifest; file path is the qualified name.
  - Workflow markdown = YAML frontmatter (typed signature) + prose + fenced ` ```step` JSON blocks. One declaration per file.
  - Type system: base scalars + `Json`, `FileRef`, `List<T>`, `Record<{...}>`, `WorkflowRef`/`ToolRef`. Full static type-check before any execution.
  - Control flow v1: sequential only. `if`/`foreach`/`par` deferred to v1.1.
  - Persistence: content-addressed step results (`step-key = hash(qname, step id, resolved args)`) + append-only `trace.jsonl`. Resume by skipping cached step keys. Chosen over pure event-sourced replay because it handles non-deterministic LLM steps cleanly.
  - Workspace sandbox: canonicalised root, path-traversal rejected, project dir read-only during exec.
  - `llm-simple` consumed as local `packages:` entry in `cabal.project`.
- Deferred to v2 (per idea.md stage 2): dynamic workflow synthesis by agents, trace introspection, skill extraction. Spec §13 lists these explicitly so v1 data model stays compatible.
- Open items still flagged in spec: markdown parser choice, exact frontmatter grammar, expression sub-language, CLI command name, run directory name.

## 2026-07-07 — Spec v1 revisions

- Replaced JSON step blocks with a small DSL: `bind <- qname(args)`, optional
  `@id`, `_ <-` for discard, explicit `return { ... }` blocks. Multi-line
  strings via `"""..."""`, string interpolation via `${...}`.
- Added `@self#<heading-slug>` expression to reference raw markdown content
  under a heading in the current file. Locked in for v1 (not deferred).
- Added ambient typed `Context` in scope for every step:
  `{ workspace, run, self, inputs, trace, env }`. Access via `${ctx.*}`.
  No implicit threading at call sites.
- Escape hatch chosen: `builtin/introspect` tool (not a `${ctx.raw}`
  expression) so usage is visible as a step in the workflow source.
- Step-key hashing now includes only a projection of `ctx` restricted to
  *stable* fields. Steps that reference volatile fields
  (`ctx.trace`, `ctx.run.started_at`) or call `builtin/introspect` are
  statically marked non-cacheable and always re-execute on resume.
  Rationale: their purpose is to observe live state; caching defeats them.
- Added `Secret<T>` type in v1. Auto-tag env vars matching
  `*_KEY|*_TOKEN|*_SECRET|*_PASSWORD`. Redacted in `trace.jsonl` as
  `<secret:$name>`. Forbidden from plain-string interpolation.
- `project.json` gains an optional `env` whitelist; nothing else from the
  process env is visible via `ctx.env`.
- Type checker factored as pure `Project -> Either [TypeError] TypedProject`
  so v1.1 dynamic-workflow evaluation can reuse it at runtime.
- Acceptance criteria expanded to A1–A9 to cover the new behaviour.

## 2026-07-07 — Revised: skip `LLM.Load`, own key store

Reconsidered the `llm-simple` `Load` decision after user push-back.
`LLM.Load.loadGateways` combines three concerns we want separated:
`.env` discovery via cwd, key lookup from hardcoded env-var names, and
gateway construction. Using it forced us into:

- Chdir at startup (surprising, affects unrelated CLI path resolution).
- `.env` pinned to project dir (couples secrets to source tree).
- Partial `error` + silent `catch SomeException` in the wrappers.
- Inconsistency with our own `Secret<T>` discipline (keys enter as
  opaque closures, never as typed secrets).

**New design (spec §7.2, §7.3 rewritten):**

- `hwfi` uses `LLM.Providers.*` gateway constructors and
  `LLM.Load.ModelCatalog.loadModelCatalog` directly. Skips
  `LLM.Load.loadGateways` and the `*OrThrow` model loaders.
- Own `KeyStore :: Map ProviderName (Secret Text)`, populated by
  merging (highest first) `--env-file`, `<project>/.env`,
  process environment. Uses `Configuration.Dotenv.parseFile` (no
  process-env injection). No chdir.
- Every project **must** ship its own `model-catalog.json`. No engine
  default. Simpler, more explicit, avoids "our default changed"
  regressions. Dropped backlog task 6.8 (catalog merging); replaced
  with 6.8 for a user-level `$XDG_CONFIG_HOME/hwfi/.env` fallback.
- Provider–key linkage validated at startup against catalog contents.
  Missing key fails immediately with a message naming the offending
  model and provider (new A12).
- Added `--env-file` CLI flag (§9). Removed the cwd/chdir language and
  the "engine default catalog" language from §7.
- TASKS 1.6 rewritten as `Hwfi.Runtime.KeyStore`; 1.7 rewritten to
  require project-local catalog; 4.5 rewritten as
  `Hwfi.Runtime.Gateways` on top of `LLM.Providers.*`.

## 2026-07-07 — Spec review fixes (pre-M1 hardening)

Addressed a review (temporary `issues.md`) before starting M1.

- **Cache invalidation on code edits (critical).** Step-key now includes
  a `callee-fingerprint`: a Merkle `fingerprint(d)` over the acyclic
  direct call graph (normalized AST + sorted callee fingerprints).
  Editing a called tool/sub-workflow (or anything transitive) now
  correctly busts dependent cached steps across abort/resume.
  `WorkflowRef`/`ToolRef` args contribute the referenced fingerprint,
  not just the qname. Spec §8.1; A13.
- **Missing env vars.** Decided **strict presence** (§5.7): an absent
  whitelisted `env` var aborts startup before any step. Keeps
  `ctx.env.X : String` total; avoids introducing `Optional<T>` in v1
  (deferred, §13). A14.
- **`TypeExpr` alias reference (bug).** Added `| QName` to the grammar
  and documented alias resolution + the deliberate `:` (types) vs `=`
  (values) split. §3.4.
- **Multi-turn chat.** Added `builtin/llm-chat` with a `messages`
  history (message-based `GenRequest`). Aligns with the agentic stage-2
  goal. §6; A16.
- **Interpolation of complex types.** Defined bare-ref vs in-string
  positions (§3.2.1) and a total rendering table (structured → compact
  JSON; `Bytes`/`Secret` rejected at check time). No runtime type errors
  from interpolation.
- **`ctx.trace` on resume.** Removed the earlier synthetic
  `StepStart/StepEnd(cached=true)` events (they made observed history
  depend on caching). New model: cached steps emit no new events; their
  original detailed events persist in `trace.jsonl`; `ctx.trace` is the
  full parse of the file (§8.3.5). Added a `Resumed` marker; `RunEnd`
  now only `completed|aborted` and appears once at logical-run end.
  Reworked ordering invariants for multi-attempt runs. A15.
- **Eval errors.** Added `eval` error kind for list OOB / missing `Json`
  fields / bad interpolation targets (§8.3.2). Documented that field
  access on opaque `Json` and list indexing are not statically checked
  (§5.6.7).
- **Return rule.** Tightened §5.6.5: explicit `return` required when
  `outputs` is non-empty, unless the last step's result type is
  structurally equal to `outputs`.
- TASKS updated: added 2.6b, 3.9 (interpolation/return), 3.10
  (fingerprinting), 4.6 (evaluator + eval errors), 5.5 (`ctx.trace`
  reconstruction); renumbered following items. Added A13–A16.

  be quoted in frontmatter (`"Record<{ role: String }>"`). Spec examples

## 2026-07-07 — M3 type checker complete

- Milestone: `hwfi check` type-checks a whole project into a
  `TypedProject` (resolved signatures, per-step cacheability, Merkle
  fingerprints). 71 tests green across unit specs and expected-error
  fixtures (`test/fixtures/check/`).
- Decision: split the type world in two — the surface `TypeExpr` (AST,
  unresolved, alias refs allowed) vs `Hwfi.Type.Type` (resolved, used by
  the checker). Aliases are expanded during resolution; cycles are an
  error (A10).
- Decision: assignability is `structEq` plus one subtyping rule — a
  `String` is accepted where a `FileRef` is expected — so literal path
  strings can feed file-taking parameters without a cast. Records compare
  structurally (field order irrelevant); lists/secrets are congruent.
- Decision: declaration `Fingerprint` is a Merkle hash (crypton SHA256)
  over the canonical AST encoding plus the sorted callee fingerprints,
  taken over the acyclic direct call graph. Builtins are leaves with a
  fixed identity; bare first-class refs are not static callees.
- Blocker resolved (root cause): `computeFingerprints` hung on any
  multi-decl project. Cause: `Fingerprint` is a `newtype`, so forcing a
  value to WHNF forces its hash and therefore its callee lookups into the
  self-referential `fps` map; `Data.Map.Strict.mapWithKey` forces every
  value during construction, demanding a callee's fingerprint before
  `fps` exists → loop. Fix: build `fps` with `Data.Map.Lazy.mapWithKey`
  (tie-the-knot), leaving lookups on the shared map type unchanged.
- Decision: checker is pure (`Project -> Either [TypeError] TypedProject`),
  keeping IO in the CLI only — required for the deferred v1.1 dynamic
  workflow evaluation.

## 2026-07-07 — M4 runtime and built-in tools complete

- Milestone: `hwfi run` executes a checked project end-to-end. New modules
  `Hwfi.Runtime.{Value,Error,Trace,Workspace,Gateways,Context,Eval,Builtins,
  Executor}`. 102 tests green (was 71); new deps `time`, `random`, `vector`,
  `unliftio`. `examples/summarise/` added.
- Decision: a dedicated `RValue` runtime type, richer than aeson `Value`,
  so the executor keeps the distinctions JSON conflates — `FileRef` vs
  `String`, `Secret<_>` (redacted, carries source name) vs its payload, and
  first-class `ToolRef`/`WorkflowRef` values vs a qname string. The checker's
  static types tell the executor which constructor to build.
- Decision: interpolation of structured values and step-key hashing share
  one canonical JSON encoder (sorted object keys, compact) in
  `Hwfi.Runtime.Value`, so §3.2.1 rendering and §8.1 hashing can never drift.
- Decision: the trace event schema (§8.3) and an in-memory `Tracer` were
  built now, in M4, not deferred to M5 — `ctx.trace` and `builtin/introspect`
  both need live events. M5 layers the `trace.jsonl` writer + resume
  reconstruction on the same `emit` seam rather than redefining the shape.
- Decision: workspace traversal guard is *lexical* (collapse `.`/`..`, reject
  escapes), not `canonicalizePath`-based, so a symlink created mid-run can
  never redirect a `FileRef` outside the root (A5). Root is canonicalised
  once at startup.
- Decision: gateways built from `LLM.Providers.*` + our typed `KeyStore`
  (never `LLM.Load.loadGateways`, per §7.2); `ollama` is always present
  (no key). `ModelConfig` assembly mirrors `LLM.Load.LoadModels` but sources
  keys from the `KeyStore`. Unknown `model` args fail listing available
  names (A11).
- Decision: `ctx` is rebuilt per step because `self.step_id` and `trace`
  vary; the rest comes from an immutable `RunInfo`. `ctx.inputs` is always
  the *root* inputs (even inside sub-workflows), while the `inputs` binding
  is the current declaration's own inputs.
- Decision: errors nest — every failing step in a call chain emits its own
  `error` event tagged with its `(qname, step_id)`, so each `StepStart` has
  exactly one terminal (§8.3.3 invariant 3). The returned `RuntimeError`
  keeps the deepest step location for the CLI message.
- Deferred to M5 (as planned): `.hwfi/runs/<id>/`, `run.json`, `trace.jsonl`
  persistence, step-key caching + resume, workspace lock, `hwfi show`. The
  `run-start` `project_hash` is a fingerprint stand-in until M5's project-dir
  content hash.
- Note: `builtin/llm-chat` maps `tool`-role messages to a user turn for now;
  full `ToolResult` round-tripping (call ids) is a stage-2 concern.
